## HTTP 版本

### HTTP 1.0
连接不能复用，每次http都要重新进行tcp连接 
- 占用资源
- 无法充分利用资源 (TCP)
- 连接是阻塞的

### HTTP 1.1 
增加了管道化，多个HTTP请求可以使用一个TCP连接，发送是非阻塞的，可以一次性发多个http请求
但是接受必须按顺序 -> 队头阻塞 不然就无法区分响应对应哪个http连接


### HTTP 2.0
- 二进制分帧 2.0 引入了 二进制分帧 对每条流都设置一个流编号，在二进制层面，将http请求拆分成多个帧，每个帧都包含 流编号 长度等信息，通过这种方式，可以解决http层面上对头阻塞的问题，不同http请求可以乱序发送 全双工

但是 也无法解决tcp层面上的 队头阻塞 => http3 QUICK协议 UDP协议

- 头部压缩 
  使用静态表 记录常用 固定的请求头 用一个字节 表示头编号 规范里有1-61 61个
  动态表 记录其他的请求头，通讯双方严格同步动态表，index从61开始 理论无上限！
  但是双方会协商动态表总长度 一般是 4096B

  对于 值，使用哈夫曼编码压缩长度 可以用平均5字节表示8字节的长度

- 服务器推 SSE SSE（Server-Sent Events）必须先建立一个 HTTP 连接，然后在这个连接上持续推送数据。 （MCP用到）

### HTTP3 QUICK协议
解决了 TCP层面的队头阻塞 使用UDP协议
